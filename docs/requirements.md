# 프론트엔드 요구사항 분석

## 프로젝트 목적

* 사용자가 질문 게시글을 작성하고 조회할 수 있는 Q&A 기반 서비스 화면을 제공한다.
* MVP 단계에서는 회원가입/로그인 화면, 카테고리 조회, 게시글 작성, 게시글 목록 조회 화면을 우선 구현한다.
* 백엔드 API와 연동하여 사용자가 브라우저에서 실제 기능 흐름을 확인할 수 있도록 한다.

---

## 사용자 유형

### 비로그인 사용자

* 회원가입 페이지로 이동할 수 있다.
* 로그인 페이지로 이동할 수 있다.
* 카테고리 목록을 조회할 수 있다.
* 게시글 목록을 조회할 수 있다.

### 일반 사용자

* 게시글 작성 페이지로 이동할 수 있다.
* 카테고리를 선택해 게시글을 작성할 수 있다.
* 게시글 목록을 조회할 수 있다.
* 게시글 상세 페이지로 이동할 수 있다.

### 관리자

* MVP 범위에서는 별도 관리자 화면을 구현하지 않는다.

* `/posts/create` : 게시글 작성

### [x] 게시글 목록 페이지

* 사용자는 게시글 목록을 조회할 수 있다.
* 게시글 목록 페이지 진입 시 백엔드 게시글 목록 조회 API를 호출한다.
* 목록에는 게시글 ID, 제목, 작성자 닉네임, 새 글 여부, 생성일을 표시할 수 있다.
* 게시글 목록 응답의 `newPost` 값이 `true`인 게시글은 제목 또는 목록 항목 근처에 작은 `new` 표시를 노출한다.
* 게시글 목록 응답의 `newPost` 값이 `false`인 게시글에는 `new` 표시를 노출하지 않는다.
* 현재 MVP에서는 우선 제목을 표시하고, 제목 클릭 시 상세 페이지로 이동한다.
* 게시글이 없으면 “게시글이 없습니다.” 메시지를 표시한다.
* API 요청 중에는 로딩 상태를 표시한다.
* API 요청 실패 시 에러 메시지를 표시한다.
* “새 글 작성” 버튼 클릭 시 게시글 작성 페이지로 이동한다.

#### 사용 API

* `GET /api/posts`

#### 프론트 처리

* `getPosts()` API 함수 사용
* 백엔드 `ApiResponse` 구조에 따라 실제 목록 데이터는 `response.data.data`에서 추출
* `postList` 상태로 목록 관리
* `loading`, `error` 상태 처리

### [x] 게시글 작성 페이지

* 사용자는 카테고리, 제목, 내용을 입력해 게시글을 작성할 수 있다.
* 게시글 작성 페이지 진입 시 카테고리 목록 조회 API를 호출한다.
* 카테고리명은 select 박스에 표시한다.
* 사용자가 선택한 카테고리의 `id`를 게시글 작성 요청의 `categoryId`로 전달한다.
* 제목은 필수이며 100자 이하이다.
* 내용은 필수이며 5000자 이하이다.
* 제목과 내용은 공백만 입력할 수 없다.
* 입력값 검증 실패 시 API 요청을 보내지 않고 경고 메시지를 표시한다.
* 게시글 작성 성공 시 성공 메시지를 표시하고 게시글 목록 페이지로 이동한다.
* 게시글 작성 실패 시 서버 에러 메시지를 표시한다.

#### 사용 API

* `GET /api/categories`
* `POST /api/posts`

#### 요청 데이터

```json
{
  "userId": 1,
  "categoryId": 1,
  "title": "제목",
  "content": "내용"
}
```

#### 프론트 처리

* `categoryList` 상태로 카테고리 목록 관리
* `categoryId` 상태로 선택된 카테고리 관리
* `title`, `content` 상태로 입력값 관리
* `validateForm(title, content, categoryId)`로 입력값 검증
* `createPost(postData)` API 함수 사용
* 성공 시 `/posts`로 이동

### [ ] 게시글 상세 페이지

* 사용자는 게시글 목록에서 제목을 클릭해 상세 페이지로 이동할 수 있다.
* URL의 `postId`를 기준으로 게시글 상세 정보를 조회한다.
* 게시글 제목, 내용, 작성자 닉네임, 생성일, 카테고리 정보를 표시할 수 있다.
* API 요청 중에는 로딩 상태를 표시한다.
* 존재하지 않는 게시글이면 에러 메시지를 표시한다.
* 수정 버튼 클릭 시 게시글 수정 페이지로 이동한다.

#### 사용 예정 API

* `GET /api/posts/{postId}`

#### 프론트 처리 예정

* `useParams()`로 `postId` 추출
* `getPost(postId)` API 함수 사용
* `post`, `loading`, `error` 상태 관리
* `useEffect`에서 `fetchPostDetail()` 실행
* `postId` 변경 시 재조회

---

### [ ] 게시글 수정 페이지

* 사용자는 기존 게시글의 카테고리, 제목, 내용을 수정할 수 있다.
* 수정 페이지 진입 시 기존 게시글 상세 정보를 조회한다.
* 카테고리 목록도 함께 조회해 select 박스에 표시한다.
* 기존 카테고리가 선택된 상태로 표시되어야 한다.
* 제목은 필수이며 100자 이하이다.
* 내용은 필수이며 5000자 이하이다.
* 제목과 내용은 공백만 입력할 수 없다.
* 입력값 검증은 Form 컴포넌트와 PostEditPage 제출 처리 양쪽에서 수행한다.
* 입력값 검증 실패 시 API 요청을 보내지 않고 경고 메시지를 표시한다.
* 수정 버튼 클릭 시 `PATCH /api/posts/{postId}` 요청을 보낸다.
* 수정 요청 중에는 `isLoading` 상태로 화면을 제어한다.
* 수정 실패 시 `error` 상태를 설정하고 서버 에러 메시지를 우선 표시한다.
* 수정 성공 시 해당 게시글 상세 페이지(`/posts/{postId}`)로 이동한다.

#### 사용 예정 API

* `GET /api/posts/{postId}`
* `GET /api/categories`
* `PATCH /api/posts/{postId}`

### [ ] 회원가입 페이지

* 사용자는 이메일, 비밀번호, 닉네임을 입력해 회원가입할 수 있다.
* 이메일은 필수이며 이메일 형식이어야 하고 중복될 수 없다.
* 비밀번호는 필수이며 8자 이상이어야 한다.
* 닉네임은 필수이며 2자 이상 20자 이하이고 중복될 수 없다.
* 입력값 검증은 Form 컴포넌트와 Page 제출 처리 양쪽에서 수행한다.
* 입력값 검증 실패 시 API 요청을 보내지 않고 경고 메시지를 표시한다.
* 회원가입 요청 중에는 `isLoading` 상태로 화면을 제어한다.
* 회원가입 실패 시 `error` 상태를 설정하고 회원가입 페이지에 그대로 머무른다.
* 회원가입 실패 시 알림창으로 "회원가입 실패" 메시지를 한 번 표시한다.
* 회원가입 성공 시 게시글 목록 페이지로 이동한다.

#### 사용 예정 API

* `POST /api/auth/signup`

## 제외 범위

* MVP에서는 JWT 발급/검증 및 토큰 저장 흐름을 본격적으로 구현하지 않는다.
* MVP에서는 관리자 전용 화면을 구현하지 않는다.
* MVP에서는 댓글, 검색, 페이징, 좋아요, 파일 업로드를 구현하지 않는다.

* MVP에서는 디자인 시스템 수준의 공통 UI 컴포넌트까지 만들지 않는다.

## 프론트 구현 기준

### API 통신

* `axios` 기반으로 API 요청을 처리한다.
* 공통 axios 설정은 `api.js`에 둔다.
* 도메인별 API 함수는 별도 파일로 분리한다.

  * `postApi.js`
  * `postCategoryApi.js`
  * 추후 `authApi.js`

### 백엔드 공통 응답 처리

백엔드 성공 응답은 `ApiResponse` 구조를 따른다.

```json
{
  "status": 200,
  "message": "요청 성공 메시지",
  "data": {}
}
```

프론트에서는 API 함수에서 필요한 형태로 응답을 정리한다.

* 목록/상세 조회처럼 실제 데이터만 필요한 경우: `response.data.data`
* 작성/수정/삭제처럼 성공 메시지도 필요한 경우: `response.data`

### 에러 처리

* API 요청 실패 시 `catch`에서 처리한다.
* 서버 에러 메시지는 `error.response?.data?.message`에서 가져온다.
* 서버 응답이 없을 경우 기본 에러 메시지를 사용한다.

### 상태 처리

각 페이지는 필요한 경우 다음 상태를 관리한다.

* `loading` 
* `error`
* `empty`
* `submitting`
* form 입력 상태

### 검증 처리

* 프론트 검증은 사용자 편의용 1차 검증이다.
* 백엔드 검증은 최종 방어선이다.
* 게시글 작성/수정에서는 `postValidation.js`의 검증 함수를 사용한다.
